# AI 应用工程师 JD 样本模板

> 这个模板的重点不是“把岗位写得很大”，而是把 AI 应用工程师最容易被误解的部分写清楚：业务场景、交付边界、上线标准、评测方式。

## 1. 岗位基本信息
- 公司名称：`[公司名]`
- 业务方向：`[例如：企业知识工作台 / 客服助手 / 销售助手 / 内部流程助手]`
- 岗位名称：AI 应用工程师
- 汇报对象：`[产品负责人 / 技术负责人 / 创始人]`
- 工作地点：`[城市 / 远程 / 混合]`
- 经验要求：`[X- Y 年]`
- 用人目的：`[新项目落地 / 存量系统优化 / 业务提效]`

## 2. 岗位目标
在明确业务场景和约束条件下，基于大模型、RAG、工具调用和工作流编排，交付可上线、可评测、可持续优化的 AI 应用。

## 3. 业务场景描述
- 当前要解决的业务问题：`[一句话说明]`
- 用户是谁：`[内部员工 / 客户 / 销售 / 客服 / 运营]`
- 主要使用场景：`[问答 / 检索 / 总结 / 生成 / 规划 / 处理工单]`
- 成功标准：`[效率提升、准确率、响应时延、人工替代比例等]`
- 不做什么：`[明确边界]`

## 4. 核心职责
- 负责 AI 应用的需求拆解、技术方案设计与落地实现
- 处理 Prompt、上下文管理、工具调用、知识库检索、工作流编排
- 设计评测、日志、反馈回流和迭代优化机制
- 与业务方协作，把“能演示”做成“能稳定用”
- 控制效果、成本、延迟和异常兜底

## 5. 必须项
- 有 LLM 应用实战经验，不只是调用 API
- 理解 Prompt、RAG、上下文窗口、工具调用的基本原理
- 具备 Python / Node.js 至少一种工程能力
- 能把业务问题拆成可执行流程
- 有调试效果、控制成本、处理异常的经验
- 能说明自己做过的项目如何上线、如何评测、如何迭代

## 6. 加分项
- 有 Dify / n8n / LangGraph / OpenAI SDK / MCP 等实战经验
- 有评测、日志、反馈回流、数据清洗经验
- 有 AI Coding、企业知识库、客服、销售场景经验
- 理解 embedding、rerank、chunking，但不会只停留在概念

## 7. 风险项 / 红旗
- 只会做 demo，不会做稳定系统
- 过度依赖单一模型，没有 fallback 思维
- 只讲技术术语，不讲业务结果
- 不理解成本、效果、延迟之间的平衡
- 没有任何可验证的项目证据

## 8. 评估重点
- 做过的 AI 应用解决了什么业务问题
- 知识库 / RAG / workflow 是怎么设计的
- 效果不稳定时如何排查
- 如何做 prompt、工作流、反馈闭环优化
- 如何证明“上线可用”而不只是“demo 可看”

## 9. 试用期或入职后 30 天交付
- 输出 1 个可上线的 AI 场景原型
- 建立基础日志和反馈记录方式
- 明确 1 套评测指标
- 完成 1 次基于反馈的迭代

## 10. 填写示例
- 公司名称：`星链知识工作台`
- 业务方向：企业内部知识问答 + 工单总结
- 汇报对象：产品负责人
- 用人目的：在 6 周内落地可用的内部知识助手
- 成功标准：
  - 一线员工检索答案时间减少 50%
  - 常见问答命中率提升到可用水平
  - 复杂问题可转人工兜底

### 示例职责
- 接入内部知识库，完成 RAG 检索和回答生成
- 设计工具调用流程，支持查询工单、拉取文档、生成摘要
- 建立人工纠错与反馈回流机制
- 对线上结果做抽样评测并持续优化

### 示例必须项
- 至少 1 个 AI 应用从 0 到 1 上线经验
- 能说明如何做 chunking、rerank、prompt 调整和 fallback
- 能说明如何记录日志、评测效果和处理失败案例
